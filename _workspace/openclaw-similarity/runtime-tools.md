# Runtime & Tools Comparison

## 한 줄 결론

**유사도 ≈ 47%** — 핵심 ReAct 루프·tool 입력 버퍼링·스트리밍 FSM·프롬프트 캐싱·context guard/compaction 의 "내부 단일 backend 부분"은 충실히 모방했으나, OpenClaw 의 본질적 정체성인 **다중 backend(claude-cli/codex-cli + 8+ provider) · plugin-sdk · 80+ skill 카탈로그 · 컨테이너 기반 bash/sandbox · 자기-편집(apply-patch) · 멀티모달 · cache-trace · tool-loop-detection · failover/auth-profile cooldown 의 깊이** 는 대부분 누락 또는 적응 형태로 축소.

## 카테고리별 분석

### Faithful (충실 모방) — 9개

- **Compaction** (`packages/agent/src/agents/context/compaction.ts:11-36, 178-179`): SAFETY_MARGIN=1.2, SUMMARIZATION_OVERHEAD_TOKENS=4096 등 OpenClaw 상수를 그대로 채택, 3-단계 폴백(summarize-full → summarize-partial → truncate-oldest)도 동일 모델.
- **Context window guard** (`window-guard.ts:32-35,80-99`): hard-min 16K/standard 32K, warn 70%/critical 85%, reserveTokens=4096 — OpenClaw 의 `CONTEXT_WINDOW_HARD_MIN_TOKENS=16_000, CONTEXT_WINDOW_WARN_BELOW_TOKENS=32_000` (`context-window-guard.ts:3-4`)을 도메인 맞게 변형 채택.
- **Tool input buffer** (`tool-input-buffer.ts:23-50`): Anthropic stream 의 `tool_use_start → tool_input_delta → tool_use_end` 시퀀스 조립 패턴이 SDK 표준과 일치. OpenClaw 는 cli-runner 내부에서 같은 모델로 처리.
- **Tool registry + policy 9 stage** (`registry.ts`, `policy.ts:215-227`): OpenClaw 의 분산된 tool-policy / tool-mutation / tool-loop-detection 을 단일 클래스로 합쳐 9 stage 파이프라인으로 형식화. 기능적으로 동등.
- **Auto-reply 파이프라인 stage 분리** (`pipeline.ts:78-319`): OpenClaw 가 100+ 파일에 분산한 stage 사상을 6+2 stage로 명시화. 핵심 사상 충실.
- **Prompt caching `cache_control` 부착** (`anthropic.ts:114-138`): system prompt 와 tools 마지막 항목에 `ephemeral` 부착. Anthropic 권장 패턴 충실.
- **Concurrency lane** (`runner.ts:75,120`): lane manager 획득/release 로 사용자별 동시성 제한.
- **Token counter 임계값** (`tokens.ts:51-61`): 80%/95% 1회 경고 발행 패턴.
- **Failover 분류 핵심** (`errors.ts:38-83`): 401/403 불-폴백, 429 rate-limit, 529 model-unavailable, 5xx server-error 등 OpenClaw 의 등급체계 핵심 충실.

### Adapted (정당한 단순화) — 6개

- **Agent loop**: OpenClaw 는 cli-runner(외부 CLI)와 SDK provider 양쪽 지원, FinClaw 는 SDK 직접 호출 단일경로. **정당화: 사용자 1인·SDK 직호출이 감사 가능성에 더 유리.**
- **Auth profile store**: OpenClaw 의 영속화·OAuth flow·repair 까지 1,500 LOC. FinClaw 는 in-memory + 핵심 모델 500 LOC. **정당화: 1인용·multi-tenant 부재.**
- **Failover error 분류 차원**: billing/format/model_not_found 누락 → FailoverError 5종으로 축소. **정당화: SDK 단일 사용. 결제·포맷 에러는 사용자가 직접 보면 됨.**
- **Plugin 시스템 (server-side)**: OpenClaw 의 1,900 LOC 분산 시스템 vs FinClaw 의 700 LOC 단일 패키지. 핵심 hook/registry/loader/manifest 모델 동등. **정당화: 1인용에 외부 plugin 배포 채널 불필요.**
- **Hooks 깊이**: OpenClaw 는 LLM 호출 전후·compaction 전후·subagent 등 ~20+ hook. FinClaw 는 11개. **정당화: 운영 자동화 기능 축소.**
- **Routing**: account-id/binding 모델 동일. 단일 사용자라 단순화. 정당.

### Diverged (의도적 차별화) — 3개

## 핵심 발견

1. **내부 1-provider 루프는 거의 완성형이고 모범적이다** — Compaction·context guard·tool registry·tool policy·token counter·prompt caching·auto-reply 파이프라인의 코드 품질은 OpenClaw 수준. 단순화의 결과가 더 깔끔하다.
2. **OpenClaw 의 정체성("multi-channel/multi-backend AI 게이트웨이")은 명백히 포기됨** — multi-provider · backend-as-CLI · plugin SDK · skill 카탈로그 · MCP · 멀티모달 모두 부재. 도메인 합체(금융 1인용)로 정당화는 일부 가능하나 plans/CLAUDE.md/use_case 의 명시 근거가 부분적이라 "근거 부족" 누락도 포함.
3. **Tool loop detection 은 Misimplemented 1건** — 흉내만 내고 효과 없음. project_use_case.md "감사 가능성 우선" 원칙과 직접 충돌. 단기 수정 필요.
4. **Cache trace 부재** — 프롬프트 캐싱은 동작하나 추적이 없어 비용 audit 가 불가능. 감사 가능성 원칙과 충돌하는 우발 누락.
5. **Multi-provider Critical missing** — Anthropic 단일 의존이 가장 큰 운영 위험. ProviderAdapter 인터페이스가 열려있어 OpenAI/Google adapter 1~2개 추가만으로 점수 점프 가능.
