# Architecture Comparison

## 한 줄 결론

**유사도 56% — 모노레포·진입점·테스트·배포의 "겉모양" 은 닮았으나, plugin-sdk / 다중-Dockerfile 샌드박스 / fly+render 다중 배포 / 5-tier 테스트 / 빌드 단계 일체가 부재. 11 패키지 분할은 OpenClaw 가 안 가진 진정한 차별화이지만 그 빈자리를 메우는 패키지 내 분할 깊이는 약함.**

## 카테고리별 분석

### Faithful (충실 모방)

- **#3 CLI bin 진입점** — 둘 다 `package.json:bin` 으로 노출. FinClaw 의 forwarder 가 6줄로 짧지만 본질 패턴은 동일. 인용:
  - OpenClaw `package.json:16-18`: `"bin": { "openclaw": "openclaw.mjs" }`
  - FinClaw `packages/server/package.json` (server 내부 bin): `"bin": { "finclaw": "./bin/finclaw.js" }`
- **#14 UI 별도 워크스페이스** — `ui/` ↔ `packages/web/`. 둘 다 vite + vitest 가진 독립 패키지.
- **#15 TUI 모듈 분리** — OpenClaw `src/tui/` 의 gateway-chat 패턴이 FinClaw `packages/tui/src/gateway-client.ts` 에 정확히 대응.
- **#20 도구 체인 (oxlint/oxfmt/tsgo/tsx/vitest)** — 동일한 next-gen Rust 기반 툴링을 채택. `package.json` devDependencies 가 거의 동일한 라인업.

### Adapted (정당한 단순화)

- **#1 pnpm workspace** — OpenClaw `(. + ui + packages/* + extensions/*)` → FinClaw `(packages/*)`. 모놀리식 src 가 없으므로 root 자체를 워크스페이스 멤버로 등록할 필요가 없음.
- **#5 daemon/process/sessions** — OpenClaw 의 OS 서비스 등록(`src/daemon/launchd.ts`, `systemd-unit.ts`, `schtasks.ts`) 은 FinClaw 가 Docker compose 로 위임. 사용자 1인·도커 실행 가정에서 정당.
- **#11 vitest 4-tier** — extensions tier 가 자연 소거. storage tier 는 SQLite 동시성 격리로 더 엄격(maxWorkers=1).
- **#16 git-hooks → lefthook** — bash 직접 작성 → lefthook 선언형 — 1인 유지보수 비용 절감.
- **#17 GitHub Actions 8→3** — 핵심(ci/deploy/release) 보존, 보조(auto-response/stale/labeler) 누락.
- **#18 scripts 78→9** — 모바일 빌드/codesign/notarize/firecrawl 등 OpenClaw 고유 작업 제거. 핵심 dev/test/setup 은 보존.

### Diverged (의도적 차별화)

- **#2 11 패키지 분할 (vs 모놀리식 src)** — _FinClaw 가 OpenClaw 보다 진짜로 깨끗한 모노레포._ OpenClaw `src/` 27 MB · 67 디렉토리는 모놀리식이며, 의존 단방향이 강제되지 않는다. FinClaw `tsconfig.json:1-16` 의 11 references 가 컴파일러 단에서 단방향 의존을 강제 (types→infra→storage→agent→...). README L17 "전체 코드를 한 사람이 읽고 수정할 수 있는 크기" 의도와 정합. **OpenClaw 의 결함을 의도적으로 고친 영역**.
- **#6 tsdown vs tsc project references** — 번들러 vs 라이브러리 컴파일. 11 패키지 인테그레이션 IDE 경험에서 tsc references 가 우월.
- **#13 brand 다중 패키지** — clawdbot/moltbot shim 부재. 단일 브랜드 정책.
- **#19 mobile apps 부재** — rubric 비대상.

### Missing (누락)

**의도된 누락 (plans/phase21/plan.md 기준):**

- **#7 plugin-sdk 별도 빌드 + exports** — plans/phase21/plan.md L78 인용:

  > "플러그인 런타임 활성화 — `packages/server/src/plugins/loader.ts` 존재하지만 호출처 없음. 별도 phase"

  현재 `packages/server/src/plugins/{discovery,errors,event-bridge,hook-types,hooks,index,loader,manifest,registry}.ts` 9 파일 골격만 존재. 외부 노출 SDK 부재. 향후 phase 에서 활성화 예정.

- **#8 extensions/\* 워크스페이스 (38개)** — 같은 plans/phase21 결정의 부산물. `extensions/` 디렉토리 자체가 root 에 없으며 `pnpm-workspace.yaml` 도 `packages/*` 만 포함. plans/phase22/plan.md L?? 의 "MCP 프로토콜, 플러그인 런타임, Evals — Phase 23+" 도 동일 의사결정 보강.
